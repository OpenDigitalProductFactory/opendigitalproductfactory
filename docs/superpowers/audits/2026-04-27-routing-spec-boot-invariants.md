## Why this exists

The routing architectural spec lists 10 boot invariants that the implementation should enforce. This audit runs each check against the *current* state of the install and reports violations *before* the spec's enforcement code lands. The output is a list of pre-existing issues that Phase A and Phase B of the migration must address.

Prior substrate attempts (per the constraint document) consistently shipped before measuring the gap they were meant to close. Running the audit on day zero is the differentiator: every violation is a tracked backlog item with a triageOutcome and effortSize before any code is written.

## Findings

### INV-6 — `BI-A58F94A2` (medium)

`ModelProfile.capabilityTier` still carries the legacy LLM-grading vocabulary in 15 rows: `'deep-thinker'` (8), `'advanced'` (4), `'moderate'` (3). Spec §8.1 requires this column to be retired (renamed to `capabilityCategory` to clarify its actual purpose as admin-UI categorization, not routing tier). The routing layer should read only `qualityTier`. Phase B of the migration is the natural place for this.

### INV-6b — `BI-6B87B3BF` (small)

Two routing source files still reference `capabilityTier`:

- `apps/web/lib/routing/known-provider-models.ts`
- `apps/web/lib/routing/recipe-types.ts`

These need an audit pass to determine whether the references are reading the value as a *routing input* (which is the bug INV-6 names) or as *admin-UI categorization* (which is fine but should be tagged for the rename). Either way, both should be addressed alongside INV-6.

### INV-9 — `BI-54411FE8` (medium)

The `anthropic-sub` provider (active subscription) is configured with `costModel='token'` and `inputPricePerMToken=0` / `outputPricePerMToken=0`. This is the misclassification documented in spec §12.2: subscription-priced providers should declare `costModel='subscription'` with a `subscriptionWindowKind`. Until that lands, the $0 token pricing means real spend (subscription quota consumed) goes uncaptured in the cost ledger — the bug Phase J just partially closed at the *token-counting* layer but cannot fully close at the *spend-attribution* layer until the subscription pricing model is added.

## What this is *not*

- This audit doesn't fix anything. It surfaces the gaps and creates trackable items.
- It doesn't validate that the spec's *implementation* is correct; that's Phase A/B testing.
- It doesn't cover the *deferred* concerns from §9 of the spec (artifact provenance, capability-derived grants, MDM, build-phase state machine). Those have their own specs and own audits.
